## 三、Critical 级别问题详情

### C-01: lib.rs 上帝文件 (1779行)

**位置**: [lib.rs 全文](../../src-tauri/src/lib.rs)
**影响**: 可维护性为零、无法单元测试、所有修改都高风险

包含内容:
- 4个数据结构定义 (Song/Artist/Album/TrackSourceInfo)
- 10个全局静态变量
- 54个函数（24个 Tauri command）
- 扫描/解析/去重/缓存全部业务逻辑
- 所有工具函数

### C-02: 双重数据模型冲突

**旧模型** ([lib.rs](../../src-tauri/src/lib.rs)): `Song`(u32 ID), `Artist`, `Album` — 实际运行时使用
**新模型** ([models/](../../src-tauri/src/models)): `Track`(String ID), `Artist`, `Album` — 标记 `#[allow(dead_code)]`

导致:
- 270+ 行手动类型转换代码 (`rebuild_memory_cache_from_persistent` + `build_persistent_cache_from_memory`)
- `music_tag/types.rs` 又定义了第三套 Artist/Album
- 向前兼容字段堆积（`ar`/`artists`, `al`/`album`, `lyric`/`lyrics` 同存）

### C-03: 多锁嵌套死锁风险

**关键位置**:
- [lib.rs:693-704](../../src-tauri/src/lib.rs#L693-L704): 同时持有 `ARTIST_SONGS_MAP` + `ALBUM_SONGS_MAP`
- [lib.rs:1012-1035](../../src-tauri/src/lib.rs#L1012-L1035): 三重锁 + 嵌套第四把锁 (`ARTIST_CACHE`)
- [lib.rs:201-220](../../src-tauri/src/lib.rs#L201-L220): 持锁期间执行文件 I/O

### C-04: unsafe std::mem::transmute 未定义行为

**位置**: 
- [parser.rs:376](../../src-tauri/src/music_tag/parser.rs#L376): APIC帧 picture_type 转换
- [parser.rs:778](../../src-tauri/src/music_tag/parser.rs#L778): FLAC图片 picture_type 转换

**风险**: 若 `u8` 值超出 `PictureType` 枚举范围(0x00-0x14)，触发未定义行为

### C-05: 音质评分计算 3+1 处重复

| # | 位置 | 函数名 | 参数形式 |
|---|------|--------|----------|
| 1 | [lib.rs:405-443](../../src-tauri/src/lib.rs#L405-L443) | `calculate_song_quality_score()` | `&Song` |
| 2 | [lib.rs:846-885](../../src-tauri/src/lib.rs#L846-L885) | `calculate_quality_score()` | 独立参数 |
| 3 | [music_library_cache.rs:102-136](../../src-tauri/src/music_library_cache.rs#L102-L136) | `TrackSource::calculate_quality_score()` | `&self` |
| 4 | [music_deduplicator.rs:275-313](../../src-tauri/src/music_deduplicator.rs#L275-L313) | `get_song_quality_score()` | `&CachedSongMetadata` |

四段代码逻辑完全一致，仅参数接口不同。

## 四、High 级别问题详情

### H-01: parse_music_file 上帝函数 (190行)

**位置**: [lib.rs:519-709](../../src-tauri/src/lib.rs#L519-L709)
承担 9 项职责: 元数据解析 → 艺术家解析 → 专辑解析 → 音质评分 → 来源构建 → Trace构建 → Song组装 → 映射更新

### H-02: rebuild_memory_cache_from_persistent 上帝函数 (180行)

**位置**: [lib.rs:974-1153](../../src-tauri/src/lib.rs#L974-L1153)
承担 8 项职责: 计数器重建 → 艺术家缓存重建 → 专辑缓存重建 → 歌曲缓存重建 → 来源信息构建 → Trace构建 → 映射更新 → 目录组织

### H-03: Tauri 命令散乱无分组

24个命令按行号分散在文件各处，同类命令相距数百行:
- 目录管理命令分布在 L888, L896, L1465, L1478, L1549
- 缓存管理命令分散在 L717, L1570, L1614, L1642

### H-04: O(n) 线性搜索性能瓶颈 (4处)

| 函数 | 操作 | 复杂度 | 优化方案 |
|------|------|--------|----------|
| [get_album_cover_data](../../src-tauri/src/lib.rs#L201-L220) | 遍历所有歌曲找封面 | O(m×n) | 建 album_id→path 索引 |
| [get_music_file](../../src-tauri/src/lib.rs#L1426-L1435) | 遍历所有歌曲找ID | O(n) | 建 song_id→path 索引 |
| [get_artist_by_id](../../src-tauri/src/lib.rs#L447-L453) | 遍历所有艺术家找ID | O(n) | 建 artist_id→Artist 索引 |
| [get_album_by_id](../../src-tauri/src/lib.rs#L456-L463) | 遍历所有专辑找ID | O(n) | 建 album_id→Album 索引 |

### H-05: 不必要的 clone() 泛滥 (30+处)

关键位置: [lib.rs:298-301](../../src-tauri/src/lib.rs#L298-L301) (克隆全部歌曲列表), [lib.rs:700,706](../../src-tauri/src/lib.rs#L700) (映射更新时反复clone Song)

### H-06: 路径遍历安全风险

- [read_cached_file](../../src-tauri/src/resource_cache.rs#L706-L714): 接受任意用户路径，可读取系统文件
- [add_music_dirs](../../src-tauri/src/lib.rs#L1550-L1557): 用户路径未验证

## 六、执行路线图

### Phase 1: 紧急修复 (P0 — 1~2周)

**目标**: 消除崩溃风险和安全漏洞，不改变架构

| 序号 | 任务 | 涉及文件 | 风险 |
|------|------|----------|------|
| 1.1 | 替换所有 `.lock().unwrap()` 为安全错误处理或 `.lock().unwrap_or_else(\|e\| e.into_inner())` | lib.rs, music_library_cache.rs, resource_cache.rs | 低 |
| 1.2 | 替换 `std::music_tag::transmute` 为 match 安全转换 | parser.rs:376,778 | 低 |
| 1.3 | 统一获取锁顺序为固定顺序（字母序），消除死锁可能 | lib.rs:693-704, 1012-1035 | 低 |
| 1.4 | 路径验证：read_cached_file 和 add_music_dirs 加入路径白名单检查 | resource_cache.rs:706, lib.rs:1550 | 低 |
| 1.5 | 提取统一音质评分函数到 common/utils.rs，删除3处重复 | lib.rs, music_library_cache.rs, music_deduplicator.rs | 低 |

**预期产出**: 消除 5 个 Critical 级问题

### Phase 2: 核心重构 (P1 — 2~4周)

**目标**: 拆分上帝文件，建立合理模块架构

| 序号 | 任务 | 涉及文件 | 风险 |
|------|------|----------|------|
| 2.1 | 创建 `common/utils.rs`: 统一 current_timestamp/is_music_file/normalize/calculate_quality_score/cache_dirs | 新文件 | 低 |
| 2.2 | 创建 `state/app_state.rs`: 封装10个全局变量为 AppState struct，使用 RwLock 替代 Mutex | lib.rs → state/ | 中 |
| 2.3 | 创建 `commands/` 目录，按功能拆分24个 Tauri command | lib.rs → commands/ | 中 |
| 2.4 | 创建 `services/scanner.rs`: 拆分 parse_music_file(190行)为 6个子函数 | lib.rs:519-709 | 中 |
| 2.5 | 创建 `services/persistence.rs`: 拆分 rebuild/save/load 为独立服务 | lib.rs:950-1263 | 中 |
| 2.6 | 建立 O(1) 索引: song_id→Path, album_id→Path, artist_id→Artist | state/app_state.rs | 中 |
| 2.7 | 删除死代码: next_song_id/artist_id/album_id, to_json() | lib.rs | 低 |

**预期产出**: lib.rs 从 1779行缩减至 ~200行（仅保留模块声明和 run()入口）

### Phase 3: 数据模型统一 (P1+ — 3~6周)

| 序号 | 任务 | 涉及文件 | 风险 |

| 3.4 | 统一 ID 类型：全面迁移至 String ID (Trace兼容) | 全局 | 高 |
| 3.5 | 消除 rebuild/build_persistent 270+行转换代码 | lib.rs | 中 |

**注意**: 此阶段需要前后端配合，建议单独开迭代

| 风险项 | 概率 | 影响 | 缓解措施 |
