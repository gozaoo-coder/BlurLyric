## 1. 现状分析

### 1.1 问题概述

项目中存在 **三套并行的数据模型**，导致以下问题：

- **代码冗余**: 约 **460 行**纯类型转换代码分散在 `lib.rs` 中
- **维护困难**: 修改一个模型需要同步修改其他两套
- **类型不安全**: 手动映射容易遗漏字段或类型错误
- **扩展受限**: 新增功能需要在三处重复实现

### 1.2 三套模型详细对比

## 2. 目标设计

### 2.1 统一后的唯一数据模型

以 `models/` 目录下的新模型为基础进行扩展，作为系统的唯一数据模型：

#### 2.1.1 核心模型层次

```
models/
├── track.rs       # Track (完整版), TrackSummup (摘要版), LocalResourceInfo
├── artist.rs      # Artist (完整版), ArtistSummup (摘要版)
├── album.rs       # Album (完整版), AlbumSummup (摘要版)
└── mod.rs         # 模块导出, 公共 Trait 定义
```

#### 2.1.2 Track 完整定义（目标状态）

```rust
/// Track - 单曲完整信息（统一后唯一模型）
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct Track {
    // ========== 基本信息 ==========
    pub id: String,                    // UUID 或自增字符串
    pub name: String,
    pub duration: Option<f64>,
    pub track_number: Option<u16>,
    pub disc_number: Option<u16>,

    // ========== 关联数据 ==========
    pub artists: Vec<ArtistSummup>,     // 艺术家摘要
    pub album: Option<AlbumSummup>,      // 专辑摘要

    // ========== 来源追踪 ==========
    pub traces: Vec<Trace>,
    pub primary_trace_index: usize,

    // ========== 本地资源 ==========
    pub local_resource: Option<LocalResourceInfo>,

    // ========== 用户偏好 ==========
    pub is_liked: bool,
    pub playlist_ids: Vec<String>,

    // ========== 扩展元数据 ==========
    pub genre: Option<String>,
    pub year: Option<u32>,
    pub comment: Option<String>,
    pub composer: Option<String>,
    pub lyricist: Option<String>,
    pub bitrate: Option<u32>,
    pub sample_rate: Option<u32>,
    pub channels: Option<u8>,
    pub lyrics: Option<String>,
    pub other_tags: Option<HashMap<String, String>>,

    // ========== 向后兼容（临时保留，最终移除）==========
    #[serde(default)]
    pub ar: Vec<LegacyArtist>,
    #[serde(default)]
    pub al: Option<LegacyAlbum>,
    #[serde(default)]
    pub lyric: String,
    #[serde(default)]
    pub src: String,
    #[serde(default)]
    pub sources: Vec<TrackSourceInfo>,
    #[serde(default)]
    pub primary_source_index: usize,
}
```

### 2.2 设计原则

1. **单一真相源**: 所有模块使用同一套模型定义
2. **渐进式迁移**: 通过 serde 兼容层保证 API 不变
3. **向前兼容**: 新字段使用 Option 类型，旧客户端忽略即可
4. **向后兼容**: 保留旧字段名，通过 rename 映射
5. **可测试**: 每个 From/TryFrom 实现都有对应单元测试

## 3. 兼容策略

### 3.1 JSON API 兼容性保证

当前前端依赖的 JSON 字段格式必须保持不变：

```javascript
// 当前前端期望的格式（不能改变）
{
  "id": 123,                    // 当前是数字
  "name": "歌曲名",
  "ar": [{ "id": 1, "name": "艺人", "alias": [] }],  // 缩写字段名
  "al": { "id": 1, "name": "专辑", "picUrl": "" },
  "lyric": "歌词内容",
  "src": "C:\\path\\to\\file.mp3",
  "traces": [...],
  "primaryTraceIndex": 0,
  "sources": [...],
  "primarySourceIndex": 0
}
```

### 3.2 Serde 兼容配置

#### 3.2.1 双字段策略（推荐）

同时输出新旧两种格式，前端逐步迁移：

```rust
#[derive(Serialize, Deserialize)]
pub struct Track {
    // 新格式（camelCase）
    #[serde(rename = "id")]
    pub id: String,

    // 旧格式兼容（数字 ID）
    #[serde(skip_serializing_if = "Option::is_none")]
    pub legacy_id: Option<u32>,

    // 新格式
    #[serde(rename = "artists")]
    pub artists: Vec<ArtistSummup>,

    // 旧格式兼容
    #[serde(rename = "ar")]
    #[serde(default)]
    pub ar: Vec<LegacyArtist>,
    
    // ... 其他字段类似
}
```

#### 3.2.2 条件序列化策略

根据请求版本决定输出格式：

```rust
impl Track {
    /// 序列化为 v1 格式（完全兼容旧前端）
    pub fn to_v1_json(&self) -> Value {
        json!({
            "id": self.parse_legacy_id(),        // 尝试解析为数字
            "name": self.name,
            "ar": self.to_legacy_artists(),
            "al": self.to_legacy_album(),
            "lyric": self.lyric.unwrap_or_default(),
            "src": self.src,
            // ...
        })
    }

    /// 序列化为 v2 格式（新前端使用）
    pub fn to_v2_json(&self) -> Value {
        json!(self)  // 直接使用 derive Serialize
    }
}
```

### 3.3 字段映射规则表

| 旧字段 (JSON) | 新字段 (Rust) | 类型变化 | 处理方式 |
|--------------|--------------|---------|---------|
| `id` (number) | `id` (string) | u32 → String | 保留数字版本作为 legacy_id |
| `ar` | `artists` | Vec<Artist> → Vec<ArtistSummup> | 同时输出两个字段 |
| `al` | `album` | Album → Option<AlbumSummup> | 同时输出两个字段 |
| `lyric` | `lyrics` | String → Option<String> | lyric 保留为默认值 |
| `src` (PathBuf) | `src` (String) | PathBuf → String | 自动转换 |
| `picUrl` | `coverUrl` | String → Option<String> | 同时输出 |

```rust
// models/mod.rs

/// 数据模型转换 trait
pub trait ModelConversion<T> {
    fn convert(self) -> T;
    fn try_convert(self) -> Result<T, ConversionError>;
}

/// 转换错误类型
#[derive(Debug)]
pub enum ConversionError {
    MissingField(String),
    TypeMismatch(String),
    InvalidId(String),
}

impl std::fmt::Display for ConversionError {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        match self {
            Self::MissingField(field) => write!(f, "Missing required field: {}", field),
            Self::TypeMismatch(msg) => write!(f, "Type mismatch: {}", msg),
            Self::InvalidId(id) => write!(f, "Invalid ID format: {}", id),
        }
    }
}

impl std::error::Error for ConversionError {}
```

## 5. ID 类型迁移策略

### 5.1 背景

当前系统使用 `u32` 自增 ID，新模型使用 `String` ID。这涉及：

- **内存中的数据结构**: HashMap 的键类型
- **持久化存储**: 缓存文件格式
- **IPC 通信**: JSON 序列化格式
- **前端 JavaScript**: ID 比较/查找逻辑

### 5.2 渐进式迁移方案

#### Phase 1: 双 ID 并存期 (当前 - v3.0.x)

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Track {
    // 新格式: String ID
    pub id: String,

    // 旧格式兼容: 数字 ID (用于前端过渡)
    #[serde(skip_serializing_if = "Option::is_none")]
    pub legacy_id: Option<u32>,

    // ...
}
```

**ID 生成规则**:

```rust
/// ID 生成器 - 支持 String 格式
pub struct IdGenerator {
    counter: AtomicU32,
    prefix: &'static str,
}

impl IdGenerator {
    /// 生成新 ID: "track_000001"
    pub fn next_id(&self) -> String {
        let num = self.counter.fetch_add(1, Ordering::SeqCst);
        format!("{}_{}", self.prefix, format!("{:06}", num))
    }

    /// 解析遗留 ID
    pub fn parse_legacy_id(id_str: &str) -> Option<u32> {
        id_str.parse::<u32>().ok()
    }
}

// 全局实例
static TRACK_ID_GEN: Lazy<IdGenerator> = Lazy::new(|| {
    IdGenerator {
        counter: AtomicU32::new(1),
        prefix: "track",
    }
});
```

#### Phase 2: 格式适配层 (v3.1.x)

**缓存格式升级**:

```rust
/// 缓存版本管理
const CACHE_VERSION_CURRENT: u32 = 2;  // 升级到 v2
const CACHE_VERSION_LEGACY: u32 = 1;   // 旧格式

impl MusicLibraryCacheData {
    /// 加载时自动升级格式
    pub fn load_with_migration(path: &Path) -> Result<Self, String> {
        let data = Self::load_from_disk(path)?;

        match data.version {
            1 => {
                // v1 → v2 迁移: u32 ID → String ID
                println!("Migrating cache from v1 to v2...");
                Ok(Self::migrate_v1_to_v2(data))
            }
            2 => Ok(data),
            v => Err(format!("Unsupported cache version: {}", v))
        }
    }

    /// 执行 v1 到 v2 的迁移
    fn migrate_v1_to_v2(v1_data: Self) -> Self {
        MusicLibraryCacheData {
            version: CACHE_VERSION_CURRENT,
            cached_at: current_timestamp(),
            songs: v1_data.songs.into_iter().map(|s| CachedSongMetadata {
                // 将 u32 ID 转为 String 格式（保持数值可解析）
                id: s.id,  // 暂时保留 u32，由上层处理
                ..s
            }).collect(),
            // ... 其他字段
        }
    }
}
```

**HashMap 键类型迁移**:

### 5.3 前端 ID 处理策略

#### 5.3.1 JavaScript 端适配

```javascript
// utils/idHelper.js

/**
 * ID 工具类 - 处理新旧 ID 格式兼容
 */
export const IdHelper = {
  /**
   * 统一 ID 比较（兼容数字和字符串）
   */
  isEqual(id1, id2) {
    const normalize = (id) => {
      if (typeof id === 'number') return String(id);
      if (typeof id === 'string') {
        // 尝试提取数字部分: "track_000123" → "123"
        const match = id.match(/(\d+)$/);
        return match ? match[1] : id;
      }
      return String(id);
    };
    return normalize(id1) === normalize(id2);
  },

  /**
   * 获取用于显示的 ID
   */
  toDisplayId(id) {
    if (typeof id === 'number') return id;
    // "track_000123" → 显示为 "123" 或完整字符串
    return id;
  },

  /**
   * 判断是否为旧格式 ID
   */
  isLegacyId(id) {
    return typeof id === 'number' || /^\d+$/.test(id);
  }
};
```

#### 6.3.1 策略一：全局数据标准化层（推荐）

创建一个统一的数据适配层，集中处理新旧格式差异：

```javascript
// api/dataNormalizer.js

/**
 * 数据标准化器
 * 将后端返回的数据统一为前端使用的格式
 */
export class DataNormalizer {
  /**
   * 标准化单曲数据
   */
  static normalizeTrack(rawTrack) {
    return {
      // ID: 统一为字符串
      id: this.normalizeId(rawTrack.id),

      // 基本信息: 直接透传
      name: rawTrack.name,
      duration: rawTrack.duration,

      // 艺术家: 优先使用新格式，fallback 到旧格式
      artists: rawTrack.artists || this.convertLegacyArtists(rawTrack.ar),
      ar: rawTrack.ar || [],  // 保留旧格式供未迁移组件使用

      // 专辑: 类似处理
      album: rawTrack.album || this.convertLegacyAlbum(rawTrack.al),
      al: rawTrack.al || null,

      // 歌词: 统一为可空
      lyrics: rawTrack.lyrics || (rawTrack.lyric || null),
      lyric: rawTrack.lyric || '',  // 保留旧格式

      // 来源路径
      src: rawTrack.src || '',

      // Trace 信息
      traces: rawTrack.traces || [],
      primaryTraceIndex: rawTrack.primaryTraceIndex ?? 0,

      // 新增字段: 直接透传
      isLiked: rawTrack.isLiked ?? false,
      playlistIds: rawTrack.playlistIds || [],

      // ... 其他字段
    };
  }

  /**
   * 标准化 ID
   */
  static normalizeId(id) {
    if (typeof id === 'string') return id;
    if (typeof id === 'number') return String(id);
    return String(id);  // fallback
  }

  /**
   * 转换旧格式艺术家
   */
  static convertLegacyArtists(legacyAr) {
    if (!legacyAr) return [];
    return legacyAr.map(artist => ({
      id: this.normalizeId(artist.id),
      name: artist.name,
      avatarUrl: null,
      aliases: artist.alias || [],
    }));
  }

  /**
   * 转换旧格式专辑
   */
  static convertLegacyAlbum(legacyAl) {
    if (!legacyAl) return null;
    return {
      id: this.normalizeId(legacyAl.id),
      name: legacyAl.name,
      coverUrl: legacyAl.picUrl || null,
      artists: [],
    };
  }
}
```

## 7. 回滚方案

### 7.1 触发回滚的条件

出现以下情况时应考虑回滚：

1. **数据丢失**: 用户报告音乐库数据损坏或丢失
2. **性能严重退化**: 启动时间增加 > 50%，内存占用翻倍
3. **前端功能故障**: 核心播放功能不可用
4. **转换错误率 > 1%**: 大量转换失败日志
5. **用户投诉激增**: 收到超过 10 个相关 bug 报告

### 7.2 回滚策略

#### 7.2.1 代码级回滚（Git Revert）

```bash
# 快速回滚到迁移前状态
git revert <migration-commit-sha>

# 或者使用 feature flag 分支
git checkout pre-migration-branch
```

**前提条件**:
- 所有更改都在独立分支上
- 主分支有稳定的 tag 可回退
- 数据库/缓存可降级

#### 7.2.2 Feature Flag 回滚（推荐）

在代码中内置开关，无需重新部署即可回滚：

```rust
// lib.rs 顶部

/// 模型迁移开关
/// true: 使用新 Track 模型
/// false: 回退到旧 Song 模型
const USE_NEW_MODEL: bool = std::env::var("BLURLYRIC_USE_NEW_MODEL")
    .map(|v| v == "1" || v == "true")
    .unwrap_or(true);  // 默认开启

// 条件编译或运行时选择
#[cfg(feature = "legacy_model")]
type CurrentTrack = Song;

#[cfg(not(feature = "legacy_model"))]
type CurrentTrack = Track;

// 或者运行时选择
macro_rules! current_track_type {
    () => {
        if USE_NEW_MODEL { Track } else { Song }
    }
}
```

**环境变量控制**:

```bash
# 强制使用旧模型（回滚）
set BLURLYRIC_USE_NEW_MODEL=0

# 使用新模型（默认）
set BLURLYRIC_USE_NEW_MODEL=1
```

#### 7.2.3 数据级回滚

**缓存数据恢复**:

```rust
impl MusicLibraryCache {
    /// 回滚到最后一次成功的缓存快照
    pub fn rollback_cache(&self) -> Result<(), String> {
        // 1. 查找备份文件
        let backup_path = self.get_backup_path();

        // 2. 验证备份完整性
        if !backup_path.exists() {
            return Err("No backup available for rollback".to_string());
        }

        // 3. 替换当前缓存文件
        let cache_file = self.get_cache_file_path();
        fs::copy(&backup_path, &cache_file)
            .map_err(|e| format!("Failed to restore backup: {}", e))?;

        // 4. 重新加载到内存
        let cache = self.load_from_disk()?;
        self.update_memory_cache(cache);

        Ok(())
    }

    /// 自动备份（每次写入前执行）
    pub fn auto_backup(&self) -> Result<(), String> {
        let cache_file = self.get_cache_file_path();
        let backup_path = self.get_backup_path();

        if cache_file.exists() {
            fs::copy(&cache_file, &backup_path)
                .map_err(|e| format!("Failed to create backup: {}", e))?;
        }

        Ok(())
    }

    fn get_backup_path(&self) -> PathBuf {
        let mut path = self.cache_dir.clone();
        path.push("music_library_cache.backup.json");
        path
    }
}
```

**ID 类型回滚**:

如果新 String ID 导致问题，快速切回 u32:

```rust
// 临时 ID 包装器
#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum CompatibleId {
    /// 新格式
    New(String),
    /// 旧格式（回滚模式）
    Legacy(u32),
}

impl CompatibleId {
    pub fn as_str(&self) -> &str {
        match self {
            Self::New(s) => s,
            Self::Legacy(n) => &n.to_string(),  // 临时转换
        }
    }

    pub fn as_u32(&self) -> Option<u32> {
        match self {
            Self::New(_) => None,
            Self::Legacy(n) => Some(*n),
        }
    }
}
```

### 7.3 回滚步骤详解

#### 场景 A: 发现严重 Bug（立即回滚）

**时间目标**: < 30 分钟

**步骤**:

1. **确认问题** (5 min)
   ```bash
   # 检查错误日志
   tail -f ~/logs/blurlyric.log | grep ERROR

   # 统计错误率
   grep "conversion error" ~/logs/blurlyric.log | wc -l
   ```

2. **启用 Feature Flag** (5 min)
   ```bash
   # 通过环境变量立即回滚
   # (如果已实现 feature flag)
   set BLURLYRIC_USE_NEW_MODEL=0
   ```

3. **重启应用** (10 min)
   ```bash
   # 重启 Tauri 应用
   taskkill /F /IM BlurLyric.exe
   start BlurLyric.exe
   ```

4. **验证回滚** (10 min)
   - 确认应用正常启动
   - 检查音乐库数据完整
   - 测试播放功能
   - 查看用户反馈

5. **通知用户** (可选)
   ```
   [公告] 我们正在修复一个问题，已暂时回退到上一版本。
   您的音乐库数据不受影响。
   ```

#### 场景 B: 性能问题（渐进回滚）

**时间目标**: < 2 小时

**步骤**:

1. **性能基线测量** (20 min)
   ```rust
   // 添加性能监控点
   let start = Instant::now();
   let _tracks = load_all_tracks();
   let duration = start.elapsed();
   log::info!("Load time: {:?}", duration);
   ```

2. **定位瓶颈** (30 min)
   ```bash

   # 检查是否有不必要的克隆
   # 检查 HashMap 查找次数
   # 检查序列化/反序列化开销
   ```

3. **优化或回滚决策** (20 min)

   决策树:
   ```
   是否能在 1h 内修复?
   ├── YES → 修复并热部署
   └── NO  → 回滚到旧模型
              └── 安排下次迭代优化
   ```

4. **执行回滚** (同场景 A)

#### 场景 C: 数据损坏（紧急回滚）

**时间目标**: < 15 分钟

**步骤**:

1. **停止写入** (立即)
   ```rust

   EMERGENCY_STOP.store(true, Ordering::SeqCst);
   ```

2. **恢复备份** (5 min)
   ```rust
   LibraryCacheManager::instance()
       .rollback_cache()
       .expect("Rollback failed!");
   ```

3. **数据校验** (5 min)
   ```rust
   // 校验恢复的数据完整性
   let stats = get_library_cache_stats();
   assert!(stats.total_songs > 0, "No songs after rollback!");
   ```

4. **通知团队** (5 min)
   - 发送警报到团队频道
   - 记录事件详情
   - 开始 root cause 分析

### 7.5 预防措施

为降低回滚概率，应在迁移过程中：

1. **灰度发布**: 先让 5% 用户使用新版本
2. **监控仪表盘**: 实时监控关键指标
3. **自动化测试**: CI/CD 必须包含回归测试
4. **金丝雀部署**: 内部测试 ≥ 48h 后再全量发布
5. **数据备份**: 每次启动自动备份缓存
6. **详细日志**: 记录所有转换操作，便于排查
