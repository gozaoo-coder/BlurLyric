## 一、并发安全问题详解

### 1.1 当前全局状态全景

当前系统使用 **10 个独立的全局静态 Mutex** 变量管理全部运行时状态：

```rust
// lib.rs:56-67 — 当前全局状态声明
static SONG_ID_COUNTER:   Lazy<Mutex<u32>>                              = Lazy::new(|| Mutex::new(0));
static ARTIST_ID_COUNTER: Lazy<Mutex<u32>>                              = Lazy::new(|| Mutex::new(0));
static ALBUM_ID_COUNTER:  Lazy<Mutex<u32>>                              = Lazy::new(|| Mutex::new(0));
static MUSIC_CACHE:       Lazy<Mutex<HashMap<PathBuf, Vec<Song>>>>     = Lazy::new(|| Mutex::new(HashMap::new()));
static ARTIST_CACHE:      Lazy<Mutex<HashMap<String, Artist>>>          = Lazy::new(|| Mutex::new(HashMap::new()));
static ALBUM_CACHE:       Lazy<Mutex<HashMap<String, Album>>>           = Lazy::new(|| Mutex::new(HashMap::new()));
static ARTIST_SONGS_MAP:  Lazy<Mutex<HashMap<u32, Vec<Song>>>>         = Lazy::new(|| Mutex::new(HashMap::new()));
static ALBUM_SONGS_MAP:   Lazy<Mutex<HashMap<u32, Vec<Song>>>          = Lazy::new(|| Mutex::new(HashMap::new()));
static MUSIC_DIRS:        Lazy<Mutex<Vec<PathBuf>>>                    = Lazy::new(|| Mutex::new(Vec::new()));
static ALWAYS_ON_TOP_STATE:Lazy<Mutex<bool>>                           = Lazy::new(|| Mutex::new(false));
```

**问题本质**: 每个变量一把独立锁，任何需要组合访问的操作都必须获取多把锁，而锁的获取顺序没有统一约定。

### 1.2 死锁风险点 A：双重锁（位置 A）

**文件位置**: [lib.rs:693-704](../../src-tauri/src/lib.rs#L693-L704)

**当前代码**:

```rust
// parse_music_file 函数末尾 —— 更新艺术家/专辑映射关系
{
    let mut artist_songs_map = ARTIST_SONGS_MAP.lock().unwrap();  // 锁1: 获取顺序 #1
    let mut album_songs_map = ALBUM_SONGS_MAP.lock().unwrap();    // 锁2: 获取顺序 #2

    for artist in &song.ar {
        artist_songs_map
            .entry(artist.id)
            .or_insert_with(Vec::new)
            .push(song.clone());  // clone 整个 Song 对象
    }

    album_songs_map
        .entry(song.al.id)
        .or_insert_with(Vec::new)
        .push(song.clone());       // 又一次 clone
}
```

**死锁场景分析**:

| 场景 | 线程1 | 线程2 | 结果 |
|------|-------|-------|------|
| 正常 | 先锁 ARTIST_SONGS_MAP → 再锁 ALBUM_SONGS_MAP | 同顺序等待 | 安全 |
| **危险** | 先锁 ARTIST_SONGS_MAP | 先锁 ALBUM_SONGS_MAP → 再等 ARTIST_SONGS_MAP | **死锁!** |
| 隐患 | 调用链间接以相反顺序获取 | 间接调用链 | 潜在死锁 |

虽然目前代码中可能不存在显式的反向获取，但:
1. 未来维护者新增代码时极易踩坑
2. `rebuild_memory_cache_from_persistent` 中确实存在三重锁嵌套（见下文），增加了复杂度
3. Rust 编译器无法在编译期检测跨函数的锁顺序问题

### 1.3 死锁风险点 B：三重锁 + 嵌套第四把锁（位置 B）

**文件位置**: [lib.rs:1012-1035](../../src-tauri/src/lib.rs#L1012-L1035) (`rebuild_memory_cache_from_persistent`)

**当前代码**:

```rust
// 重建歌曲缓存和映射关系
{
    let mut music_cache = MUSIC_CACHE.lock().unwrap();              // 锁1
    let mut artist_songs_map = ARTIST_SONGS_MAP.lock().unwrap();    // 锁2
    let mut album_songs_map = ALBUM_SONGS_MAP.lock().unwrap();      // 锁3

    music_cache.clear();
    artist_songs_map.clear();
    album_songs_map.clear();

    let mut dir_songs: HashMap<PathBuf, Vec<Song>> = HashMap::new();

    for cached_song in &cache.songs {
        let dir = cached_song.fingerprint.path.parent()
            .map(|p| p.to_path_buf())
            .unwrap_or_default();

        // ===== 在持有 3 把锁的同时，又去获取第 4 把锁！=====
        let artists: Vec<Artist> = cached_song.artists.iter()
            .filter_map(|name| {
                ARTIST_CACHE.lock().unwrap()  // 锁4: 嵌套在锁1/2/3 内部!
                    .get(name)
                    .cloned()
            })
            .collect();

        let album = ALBUM_CACHE.lock().unwrap()  // 锁5: 又一把嵌套锁!
            .get(&cached_song.album)
            .cloned()
            .unwrap_or_else(|| Album { ... });

        // ... 构建 song 对象 ...
        for artist in &song.ar {
            artist_songs_map.entry(artist.id).or_default().push(song.clone());
        }
        album_songs_map.entry(song.al.id).or_default().push(song.clone());
        // ... 反复 clone 整个 Song 对象 ...
    }

    *music_cache = dir_songs;
}
```

**问题清单**:

| 问题 | 严重程度 | 说明 |
|------|----------|------|
| 5 层锁嵌套 | Critical | MUSIC_CACHE + ARTIST_SONGS_MAP + ALBUM_SONGS_MAP + ARTIST_CACHE(循环内) + ALBUM_CACHE(循环内) |
| 循环内反复获取锁 | High | 每首歌曲迭代都 lock/unlock ARTIST_CACHE 和 ALBUM_CACHE |
| 持锁期间做大量计算 | High | 包含 HashMap 查找、Option 处理、Vec 构建、clone 操作 |
| clone 热点 | Medium | 每个 Song 被 clone 至少 3 次（artist_map + album_map + dir_songs） |

### 1.4 死锁风险点 C：持锁期间执行文件 I/O（位置 C）

**文件位置**: [lib.rs:201-220](../../src-tauri/src/lib.rs#L201-L220) (`get_album_cover_data`) 以及 [lib.rs:1386-1404](../../src-tauri/src/lib.rs#L1386-L1404) (`get_album_cover`)

**当前代码**:

```rust
fn get_album_cover_data(album_id: u32) -> Result<Vec<u8>, String> {
    let cache = MUSIC_CACHE.lock().unwrap();  // 获取排他锁

    for songs in cache.values() {
        for song in songs {
            if song.al.id == album_id {
                // ===== 持锁期间执行文件 I/O! =====
                let parser = MetadataParser::new();
                match parser.parse(&song.src) {  // 解析音频文件: 可能耗时 10~500ms
                    Ok(metadata) => {
                        if let Some(picture) = metadata.front_cover() {
                            return Ok(picture.data.clone());
                        }
                    }
                    Err(_) => continue,
                }
            }
        }
    }
    Err("Album cover not found".to_string())
}

// get_album_cover 命令中存在完全相同的问题 (L1386-1404)
fn get_album_cover(album_id: u32) -> Result<tauri::ipc::Response, String> {
    let result = (|| {
        let cache = MUSIC_CACHE.lock().unwrap();
        for songs in cache.values() {
            for song in songs {
                if song.al.id == album_id {
                    let parser = MetadataParser::new();
                    match parser.parse(&song.src) {  // 同样持锁I/O
                        Ok(metadata) => {
                            if let Some(picture) = metadata.front_cover() {
                                return Ok(tauri::ipc::Response::new(picture.data.clone()));
                            }
                        }
                        Err(_) => continue,
                    }
                }
            }
        }
        Err("Album cover not found".into())
    })();
    result
}
```

**影响量化**:

| 操作 | 预估耗时 | 持锁阻塞影响 |
|------|----------|-------------|
| 解析 MP3 文件 (5MB) | ~50-200ms | 所有读请求阻塞 50-200ms |
| 解析 FLAC 文件 (30MB 无损) | ~200-800ms | 所有读请求阻塞 200-800ms |
| 解析失败回退到下一首歌 | 累积 | 最坏情况遍历整库 |
| 并发 get_music_list 请求 | -- | 完全串行化 |

**实际场景**: 用户打开专辑详情页触发 `get_album_cover`，同时前端轮询 `get_music_list` 刷新歌曲列表 → 后者被前者完全阻塞数百毫秒。

### 1.5 其他锁使用问题汇总

#### 1.5.1 refresh_music_cache 中的连续多锁操作

**位置**: [lib.rs:717-728](../../src-tauri/src/lib.rs#L717-L728)

```rust
#[tauri::command]
fn refresh_music_cache() -> Result<(), String> {
    *SONG_ID_COUNTER.lock().unwrap() = 0;       // 锁1
    *ARTIST_ID_COUNTER.lock().unwrap() = 0;      // 锁2
    *ALBUM_ID_COUNTER.lock().unwrap() = 0;       // 锁3

    MUSIC_CACHE.lock().unwrap().clear();         // 锁4
    ARTIST_CACHE.lock().unwrap().clear();        // 锁5
    ALBUM_CACHE.lock().unwrap().clear();         // 锁6
    ARTIST_SONGS_MAP.lock().unwrap().clear();    // 锁7
    ALBUM_SONGS_MAP.lock().unwrap().clear();     // 锁8

    let music_dirs = MUSIC_DIRS.lock().unwrap(); // 锁9
    // ...
}
```

8 次独立的 `lock().unwrap()` 调用，每次获取和释放之间存在间隙，其他线程可在间隙中插入操作导致状态不一致。

#### 1.5.2 build_persistent_cache_from_memory 中的分块锁模式

**位置**: [lib.rs:1173-1263](../../src-tauri/src/lib.rs#L1173-L1263)

```rust
fn build_persistent_cache_from_memory() -> MusicLibraryCacheData {
    // 分块1: 艺术家
    { let artist_cache = ARTIST_CACHE.lock().unwrap(); /* ... */ }  // 锁后释放
    // 分块2: 专辑
    { let album_cache = ALBUM_CACHE.lock().unwrap(); /* ... */ }    // 锁后释放
    // 分块3: 歌曲
    { let music_cache = MUSIC_CACHE.lock().unwrap(); /* ... */ }    // 锁后释放
    // 分块4: 艺术家映射
    { let artist_songs_map = ARTIST_SONGS_MAP.lock().unwrap(); /* ... */ }
    // 分块5: 专辑映射
    { let album_songs_map = ALBUM_SONGS_MAP.lock().unwrap(); /* ... */ }
}
```

5 次分块加锁之间存在时间窗口，在此窗口内其他线程可能修改数据，导致持久化的快照不一致（例如：艺术家缓存已保存但歌曲缓存尚未保存时，有新歌曲添加）。

## 二、修复方案设计

### 2.1 方案对比

| 方案 | 死锁防护 | 改动量 | 性能提升 | 推荐度 |
|------|----------|--------|----------|--------|
| A: 统一锁顺序约定 | 仅防死锁 | 小 | 无 | 临时方案 |
| **B: AppState 聚合 + RwLock** | 根治死锁 | 中 | 高 | **推荐** |
| C: 减少锁持有时间 | 辅助优化 | 小 | 中 | 必须配合B |

**最终选择**: 方案 B（主）+ 方案 C（辅）。下面详细展开。

### 2.2 AppState 结构体完整设计

#### 2.2.1 核心结构定义

```rust

/// 全局 ID 计数器组 —— 使用 AtomicU32 实现无锁递增
#[derive(Debug)]
pub struct IdCounters {
    pub song: AtomicU32,
    pub artist: AtomicU32,
    pub album: AtomicU32,
}

impl IdCounters {
    pub fn new() -> Self {
        Self {
            song: AtomicU32::new(0),
            artist: AtomicU32::new(0),
            album: AtomicU32::new(0),
        }
    }

    /// 无锁获取下一个 ID
    pub fn next_song_id(&self) -> u32 {
        self.song.fetch_add(1, Ordering::Relaxed) + 1
    }

    pub fn next_artist_id(&self) -> u32 {
        self.artist.fetch_add(1, Ordering::Relaxed) + 1
    }

    pub fn next_album_id(&self) -> u32 {
        self.album.fetch_add(1, Ordering::Relaxed) + 1
    }

    /// 重置所有计数器（用于 refresh 操作）
    pub fn reset(&self) {
        self.song.store(0, Ordering::Relaxed);
        self.artist.store(0, Ordering::Relaxed);
        self.album.store(0, Ordering::Relaxed);
    }
}

/// 专辑索引: album_id → Album 对象
```

**与旧架构对比**:

| 维度 | 旧架构 (10 x Mutex) | 新架构 (AppState) |
|------|---------------------|-------------------|
| 锁数量 | 10 个独立 Mutex | 2 个 RwLock + 3 个 AtomicU32 |
| 死锁可能 | 高（多锁嵌套） | 不可能（单一锁或无锁） |
| 读并发 | 否（Mutex 排他） | 是（RwLock 共享读） |
| ID 生成 | 需要锁 | 无锁（AtomicU32） |
| 数据一致性 | 分块锁有时间窗口 | 单锁保证原子性 |

## 三、O(1) 索引设计与 CRUD 接口

### 3.1 索引数据结构

四个索引均定义为 `LibraryState` 的字段，与主存储同步生命周期:

| 索引名 | 类型 | Key | Value | 服务查询 |
|--------|------|-----|-------|----------|
| `song_path_index` | `HashMap<u32, PathBuf>` | song_id | 文件绝对路径 | `get_music_file(song_id)` |
| `album_cover_index` | `HashMap<u32, PathBuf>` | album_id | 含封面的歌曲路径 | `get_album_cover_data(album_id)` |
| `artist_index` | `HashMap<u32, Artist>` | artist_id | Artist 完整对象 | `get_artist_by_id(artist_id)` |
| `album_index` | `HashMap<u32, Album>` | album_id | Album 完整对象 | `get_album_by_id(album_id)` |

### 3.2 索引维护接口设计

以下接口作为 `LibraryState` 的方法实现，确保索引与主存储的一致性:

```rust
// === 索引维护方法（impl LibraryState）===

impl LibraryState {
    // ==================== CREATE ====================

    /// 添加一首歌曲到 music_cache 并同步更新所有相关索引
    ///
    /// 调用时机: parse_music_file 成功后
    /// 锁要求: 需要 &mut self（即 write lock）
    pub fn add_song(&mut self, dir: PathBuf, song: Song) {
        // 1. 更新 song_path_index
        self.song_path_index.insert(song.id, song.src.clone());

        // 2. 更新 album_cover_index（如果该专辑尚无封面记录）
        self.album_cover_index
            .entry(song.al.id)
            .or_insert_with(|| song.src.clone());

        // 3. 更新 artist_index
        for artist in &song.ar {
            self.artist_index.insert(artist.id, artist.clone());
        }

        // 4. 更新 album_index
        self.album_index.insert(song.al.id, song.al.clone());

        // 5. 更新 artist_songs_map
        for artist in &song.ar {
            self.artist_songs_map
                .entry(artist.id)
                .or_default()
                .push(song.clone());
        }

        // 6. 更新 album_songs_map
        self.album_songs_map
            .entry(song.al.id)
            .or_default()
            .push(song.clone());

        // 7. 更新主存储 music_cache
        self.music_cache
            .entry(dir)
            .or_default()
            .push(song);
    }

    // ==================== READ（O(1) 查询）====================

    /// 通过 song_id 获取文件路径 —— O(1)
    ///
    /// 替代原: get_music_file 中遍历 cache.values().flatten()
    pub fn get_song_path(&self, song_id: u32) -> Option<&PathBuf> {
        self.song_path_index.get(&song_id)
    }

    /// 通过 album_id 获取含封面的歌曲路径 —— O(1)
    ///
    /// 替代原: get_album_cover_data 中双重循环遍历
    pub fn get_album_cover_path(&self, album_id: u32) -> Option<&PathBuf> {
        self.album_cover_index.get(&album_id)
    }

    /// 通过 artist_id 获取 Artist 对象 —— O(1)
    ///
    /// 替代原: get_artist_by_id 中遍历 artist_cache.values()
    pub fn get_artist_by_id(&self, artist_id: u32) -> Option<&Artist> {
        self.artist_index.get(&artist_id)
    }

    /// 通过 album_id 获取 Album 对象 —— O(1)
    ///
    /// 替代原: get_album_by_id 中遍历 album_cache.values()
    pub fn get_album_by_id(&self, album_id: u32) -> Option<&Album> {
        self.album_index.get(&album_id)
    }

    /// 获取所有歌曲的扁平列表（用于序列化返回给前端）
    /// 注意: 此操作仍需 clone，但至少不需要遍历 HashMap 嵌套结构
    pub fn all_songs(&self) -> Vec<Song> {
        self.music_cache
            .values()
            .flat_map(|songs| songs.iter().cloned())
            .collect()
    }

    // ==================== UPDATE ====================

    /// 批量重建所有索引（从当前主存储派生）
    ///
    /// 调用时机: rebuild_memory_cache_from_persistent 完成后、refresh_music_cache 完成后
    /// 保证: 在单个 write lock 内完成，原子性切换
    pub fn rebuild_indexes(&mut self) {
        // 清空旧索引
        self.song_path_index.clear();
        self.album_cover_index.clear();
        self.artist_index.clear();
        self.album_index.clear();

        // 从 music_cache 重建 song_path_index 和 album_cover_index
        for songs in self.music_cache.values() {
            for song in songs {
                self.song_path_index.insert(song.id, song.src.clone());

                self.album_cover_index
                    .entry(song.al.id)
                    .or_insert_with(|| song.src.clone());
            }
        }

        // 从 artist_cache 重建 artist_index
        for (_, artist) in &self.artist_cache {
            self.artist_index.insert(artist.id, artist.clone());
        }

        // 从 album_cache 重建 album_index
        for (_, album) in &self.album_cache {
            self.album_index.insert(album.id, album.clone());
        }
    }

    // ==================== DELETE ====================

    /// 移除指定目录的所有歌曲及其索引条目
    ///
    /// 调用时机: remove_music_dirs 时
    pub fn remove_dir(&mut self, dir: &PathBuf) {
        if let Some(songs) = self.music_cache.remove(dir) {
            for song in songs {
                // 移除路径索引
                self.song_path_index.remove(&song.id);

                // 移除艺术家映射中的该歌曲
                for artist in &song.ar {
                    if let Some(entry) = self.artist_songs_map.get_mut(&artist.id) {
                        entry.retain(|s| s.id != song.id);
                        if entry.is_empty() {
                            self.artist_songs_map.remove(&artist.id);
                            self.artist_index.remove(&artist.id); // 艺术家无歌则移除
                        }
                    }
                }

                // 移除专辑映射中的该歌曲
                if let Some(entry) = self.album_songs_map.get_mut(&song.al.id) {
                    entry.retain(|s| s.id != song.id);
                    if entry.is_empty() {
                        self.album_songs_map.remove(&song.al.id);
                        self.album_cover_index.remove(&song.al.id); // 专辑无歌则移除封面索引
                        self.album_index.remove(&song.al.id);
                    }
                }
            }
        }
    }

    /// 清空所有数据和索引（用于 refresh 操作）
    pub fn clear_all(&mut self) {
        self.clear(); // 调用已有的 clear 方法
    }
}
```

```rust
fn rebuild_memory_cache_from_persistent(cache: &MusicLibraryCacheData) {
    // ========== 第一阶段: 无锁数据准备 ==========
    // 先收集所有需要的 artist/album 数据（不持锁）
    let mut artist_lookup: HashMap<String, Artist> = HashMap::new();
    for ca in &cache.artists {
        artist_lookup.insert(ca.name.clone(), Artist {
            id: ca.id,
            name: ca.name.clone(),
            alias: ca.alias.clone(),
        });
    }

    let mut album_lookup: HashMap<String, Album> = HashMap::new();
    for cal in &cache.albums {
        album_lookup.insert(cal.name.clone(), Album {
            id: cal.id,
            name: cal.name.clone(),
            pic_url: cal.pic_url.clone(),
        });
    }

    // 构建新的 LibraryState（纯内存操作，无锁）
    let mut new_state = LibraryState::new();
    let mut dir_songs: HashMap<PathBuf, Vec<Song>> = HashMap::new();

    for cached_song in &cache.songs {
        let dir = cached_song.fingerprint.path.parent()
            .map(|p| p.to_path_buf())
            .unwrap_or_default();

        // 使用本地 lookup 而非全局锁
        let artists: Vec<Artist> = cached_song.artists.iter()
            .filter_map(|name| artist_lookup.get(name).cloned())
            .collect();

        let album = album_lookup.get(&cached_song.album)
            .cloned()
            .unwrap_or_else(|| Album { id: 0, name: cached_song.album.clone(), pic_url: String::new() });

        // 构建 Song 对象（省略中间字段赋值...）
        let song = build_song_from_cached(cached_song, artists, album);

        // 直接写入 new_state（非全局状态，无需锁）
        new_state.add_song(dir.clone(), song);
    }

    // ========== 第二阶段: 单次写锁原子替换 ==========
    {
        let mut library = app_state().library.write().unwrap();
        *library = new_state;  // 整体替换，原子性
    }
}
```

**改进点**:
1. 5 层锁嵌套 → 0 层（准备阶段）+ 1 层（提交阶段）
2. 循环内反复获取 ARTIST_CACHE/ALBUM_CACHE 锁 → 本地 HashMap 查找
3. 每次 clone Song 3 次 → `add_song` 内部统一管理
4. 分块锁的时间窗口问题 → 单次 write lock 整体替换

## 四、性能优化详案

### 4.1 RwLock 替代 Mutex

#### 4.1.1 适用场景分析

当前系统中 **读操作远多于写操作**:

| 操作类型 | 频率 | 示例 |
|----------|------|------|
| **读（高频）** | 用户交互触发 | get_music_list, get_all_my_artists, get_all_my_albums, get_music_file, get_album_cover |
| **写（低频）** | 用户主动操作 | refresh_music_cache, add_music_dirs, remove_music_dirs, 后台增量扫描 |

典型比例: 读 : 写 ≈ **100 : 1**（用户浏览 vs 用户管理）

#### 4.1.2 性能理论对比

| 场景 | Mutex (当前) | RwLock (优化后) | 提升 |
|------|-------------|-----------------|------|
| 3 个并发 get_music_list | 串行排队，每个 ~5ms | 并行执行，每个 ~5ms | **3x 吞吐量** |
| 1 个 refresh + 2 个读请求 | 读请求被完全阻塞 | 读请求可与 refresh 前后的读并行 | **减少阻塞** |
| get_music_list + get_album_cover 同时 | 完全串行 | 可并行（均为 read lock） | **2x 吞吐量** |

#### 4.1.3 迁移规则

```rust
// 读操作: 使用 .read()
let library = app_state().library.read().unwrap();   // RwLockReadGuard
let songs = library.all_songs();                      // 共享引用，不可修改

// 写操作: 使用 .write()
let mut library = app_state().library.write().unwrap(); // RwLockWriteGuard
library.add_song(dir, song);                           // 可变引用
library.rebuild_indexes();                             // 批量更新

// 注意: RwLockWriteGuard 会阻塞所有 reader，因此写操作应尽量短
```

**写锁最小化原则**:

```rust
// ❌ 错误: 持写锁做 I/O 或重计算
let mut library = app_state().library.write().unwrap();
let data = fs::read(some_path)?;  // I/O 在持锁期间!
library.process_data(data);

// ✅ 正确: 先准备数据，再短时间持写锁
let data = fs::read(some_path)?;  // I/O 在锁外
let processed = heavy_computation(data);  // 计算在锁外
{
    let mut library = app_state().library.write().unwrap();
    library.update_with_processed(processed);  // 仅内存写入在锁内
}
```

```rust
impl LibraryState {
    pub fn add_song(&mut self, dir: PathBuf, song: Song) {
        // 索引更新仅存引用或 ID，不再 clone 整个 Song
        self.song_path_index.insert(song.id, song.src.clone());  // 只 clone PathBuf

        // 映射表中改存 song_id 而非整个 Song 对象
        for artist in &song.ar {

                .entry(artist.id)
                .or_default()
                .push(song.id);  // 存 u32 ID 而非 Song
        }

        self.album_songs_map
            .entry(song.al.id)
            .or_default()
            .push(song.id);  // 存 u32 ID 而非 Song

        // music_cache 中仍需存完整 Song（这是主存储）
        self.music_cache
            .entry(dir)
            .or_default()
            .push(song);  // move 进入，零拷贝
    }
}
```

#### 4.3.2 异步改造策略

**策略: 分层处理**

由于 `MetadataParser::parse()` 是同步 CPU 密集型操作（ID3/FLAC/Vorbis 解析），不能简单加 `.await`。正确做法是:

```rust
// === parser.rs 异步适配层 ===

impl MetadataParser {
    /// 异步版本: 在 blocking 线程池中执行解析
    pub async fn parse_async<P: AsRef<Path>>(&self, path: P) -> Result<MusicMetadata> {
        let path = path.as_ref().to_path_buf();

        // 将同步的文件读取+解析移到 tokio blocking 线程池
        tokio::task::spawn_blocking(move || {
            // 这里的 fs::read 是同步的，但在专用线程中不会阻塞 async runtime
            self.parse(&path)
        })
        .await
        .map_err(|e| error::join_error(e, &path))?
    }
}
```

**audio_properties.rs 特殊处理**:

```rust
// audio_properties.rs 中的 fs::read 读取的是完整音频文件
// 对于大文件（如 100MB WAV），应考虑流式读取

// 方案 A: 保持 spawn_blocking（简单，适用于大多数场景）
pub async fn parse_audio_properties_async(path: &Path) -> (Option<f64>, Option<u32>, Option<u32>, Option<u8>) {
    let path = path.to_path_buf();
    tokio::task::spawn_blocking(move || parse_audio_properties(&path))
        .await
        .unwrap_or((None, None, None, None))
}

// 方案 B: 流式读取头部位（高级优化，仅读取前 N 字节）
// 大多数音频格式的属性信息在文件头部，无需读取全文
pub async fn parse_audio_properties_streaming(path: &Path) -> (Option<f64>, Option<u32>, Option<u32>, Option<u8>) {
    use tokio::fs::File;
    use tokio::io::AsyncReadExt;

    let mut file = File::open(path).await.ok()?;
    let mut header = [0u8; 4096]; // 通常足够读取属性信息
    let n = file.read(&mut header).await.ok()?;

    // 基于 header 数据解析（需要修改现有解析函数接受 &[u8] 而非 Path）
    parse_audio_properties_from_header(&header[..n])
}
```

## 七、总结

### 7.1 优化收益矩阵

| 优化项 | 解决的问题 | 复杂度 | 收益 |
|--------|-----------|--------|------|
| AppState 聚合 (10→2 锁) | 死锁风险 C-03 | 中 | 消除死锁可能性 |
| RwLock 替代 Mutex | 读并发能力 | 低 | 读吞吐量 3-4x 提升 |
| O(1) 索引 (4个) | H-04 性能瓶颈 | 中 | 查询 800-2500x 加速 |
| 减少 clone (映射表存ID) | H-05 内存/CPU | 中 | 内存减少 ~60% |
| 持锁 I/O 分离 | M-07 阻塞问题 | 低 | 读延迟消除 |
| I/O 异步化 | M-07/M-08 | 中 | 不阻塞 async runtime |
| tokio::spawn 统一 | M-08 线程模型 | 低 | 生命周期管理规范化 |
| AtomicU32 计数器 | 锁粒度过细 | 低 | ID 生成零开销 |

### 7.2 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 迁移引入回归 Bug | 中 | 高 | 每步迁移后全量测试 |
| 索引与主存储不一致 | 低 | 高 | `add_song`/`remove_dir` 封装一致性 |
| RwLock 写饥饿 | 低 | 中 | 监控 write wait 时间 |
| feature flag 回滚失效 | 极低 | 高 | CI 中双模式跑测试 |
